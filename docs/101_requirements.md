# ホームメディア基盤 要件定義書

## 1. サービスの目的と概要

個人利用のホームメディア基盤。動画・画像・VR・ドキュメント等、**あらゆる種類のコンテンツ**を NAS やディスク等の多様なデータソースから取り込み、ブラウザからシームレスに閲覧・配信できる環境を構築する。コンテンツを AI へのインプットとして活用することも中核的なユースケースとして想定する。マルチスクリーン・マルチ品質対応の HLS 配信を基盤とし、**性能を最優先**に設計する。

## 2. システムの提供機能 (Functional Requirements)

### 2.1 コンテンツ閲覧機能

- **コンテンツ一覧**: 登録済みコンテンツの一覧表示（動画、画像、VR、ドキュメント等）。
- **コンテンツ検索**: キーワード・タグによる絞り込み（Elasticsearch 使用）。
- **動画再生**:
  - HLS Adaptive Bitrate Streaming（ABR）による品質の自動切り替え。
  - 手動での品質選択（1080p / 720p / 480p / 音声のみ）。
  - 360度 VR 動画はブラウザ上での視点操作。
- **画像ギャラリー**: 高解像度ビューアー。
- **ドキュメント閲覧**: PDF・テキスト・各種資料ファイルのブラウザ内ビューア（後フェーズ実装）。

### 2.2 管理機能

- **コンテンツ管理 (CMS)**: タイトル・説明・タグの編集（全コンテンツ公開のため公開/非公開切り替えは不要）。
- **サムネイル管理**: 自動生成サムネイルの確認・差し替え。
- 認証・認可は不要（個人利用のため）。

### 2.3 コンテンツ取り込みパイプライン

多様なデータソースに対応した取り込みパイプラインを順次整備する。現 Phase では Bluray 動画を対象とし、ドキュメント・その他メディアは後フェーズで追加する。

#### 2.3.1 Bluray 動画取り込み（Phase 4 実装）

- **Phase 1 - Extract（物理デバイス依存）**: Bluray ディスク → MakeMKV → NFS ストレージへの MKV 保存。
  - 1 枚のディスクから複数タイトルを個別の MKV ファイルとして抽出。
  - 取り込み完了後も MKV は NFS に永続保持（再変換の容易さを確保）。
- **Phase 2 - Transform（NFS ファイル操作）**: MKV → ffmpeg → 複数品質の HLS 生成。
  - 映像+音声: 1080p / 720p / 480p（ABR マスタープレイリスト付き）。
  - 音声のみ: AAC HLS ストリーム。
  - 低画質バリアントも作成（帯域幅の少ない環境向け）。
  - HLS セグメント・マニフェストを MinIO にアップロード。
- **Phase 3 - Load（カタログ登録）**: PostgreSQL 登録 → MongoDB 同期 → Elasticsearch インデックス登録 → サムネイル生成。

#### 2.3.2 ドキュメント・その他コンテンツ取り込み（後フェーズ）

- NAS や各種ストレージ上の PDF・テキスト・画像等のファイルを取り込み、カタログ登録する。
- 取り込んだドキュメントは Elasticsearch でフルテキスト検索可能にする。
- AI インプット活用：取り込んだコンテンツをベクトル化し、RAG 等の AI パイプラインへ連携する（詳細は後フェーズで設計）。

### 2.4 データモデル

- **discs**: Bluray ディスク 1 枚を管理するエンティティ（動画コンテンツ取り込み元）。
- **contents**: 各コンテンツ（タイトル）を管理。`content_type` で種別を識別（`video` / `image_gallery` / `vr360` / `document` 等）。
- **video_variants**: コンテンツごとの動画バリアント（品質別 HLS）を管理。
- **assets**: サムネイル・ポスター・添付ファイル等のメディアアセット。

## 3. 非機能要件 (Non-Functional Requirements)

### 3.1 パフォーマンス（最優先）

- **シーク高速化**: HLS セグメントは MinIO から直接配信。Nginx キャッシュによるエッジキャッシュを最大活用し、シーク時のバッファリングを極小化する。
- **メタデータ API 応答速度**: カタログ API は ms 単位で返却。MongoDB（読み取り最適化）を活用する（API 層のインメモリキャッシュは使用しない）。
- **検索速度**: Elasticsearch によるリアルタイム全文検索。
- **トランスコードスループット**: ffmpeg の並列処理（品質バリアントごとに独立ジョブ）で取り込み時間を短縮する。GPU は不要（時間がかかっても構わない）。
- **ABR プレーヤー**: クライアントのネットワーク状況に応じた品質自動切り替えで、途切れのない再生体験を提供する。

### 3.2 シンプルさ・運用性

- 認証・認可は不要（アクセス制御はインフラ層または LAN 限定運用で実施）。
- 全てのコンポーネントは Docker コンテナ化し、K8s でデプロイする（ローカル開発は kind を使用）。
- NFS・MinIO・PostgreSQL に障害があっても他サービスへの影響を局所化する。

### 3.3 データ保全

- Bluray から変換した MKV ファイルは NFS 上に永続保持（再変換のために必要）。
- HLS セグメント（.ts, .m3u8）は MinIO に保持（配信元）。
- PostgreSQL はコンテンツメタデータの Single Source of Truth（信頼できる情報源）。
- MongoDB は PostgreSQL のキャッシュ用読み取りレプリカ（Benthos により同期）。
- Elasticsearch は全文検索インデックス（PostgreSQL から定期 or イベント駆動で同期）。

### 3.4 拡張性（将来対応）

- チャプターマーカー（VideoTimestamps）機能: 後フェーズで対応。
- プレイリスト・グルーピング機能: 後フェーズで対応。
- シリーズ/エピソード概念: 後フェーズで対応。
- 視聴履歴・レジューム再生: 後フェーズで対応。
- ドキュメント・資料取り込みパイプライン（PDF・各種ファイル形式）: 後フェーズで対応。
- AI インプット活用（コンテンツのベクトル化・RAG 基盤連携・LLM への資料供給等）: 後フェーズで対応。
